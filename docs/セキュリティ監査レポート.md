**対象プロジェクト**: スタンドアロン型 AI駆動ナレッジ&データ活用プラットフォーム「AVITO (アビト)」  
**監査日**: 2026-06-18  
**監査責任者**: Antigravity (AI)  
**監査対象フェーズ**: Phase 4 (Step2 DB連携・最終統合段階)

---

## 1. 監査目的
本監査は、閉域網運用を前提とするローカルAIプラットフォーム「AVITO」において、機密データの外部漏洩リスクが排除されていること、およびNL2SQL（自然言語からSQLを自動生成・実行する機能）における安全なアクセス制御・改ざん防御・監査追跡機能が堅牢に実装されていることを実証・確認することを目的とします。

## 3. 各監査項目の詳細検証

### 3.1. 閉域網・外部通信制限の監査
*   **LLM接続先の検証**:
    *   `OLLAMA_BASE_URL` は `http://ollama:11434` に設定されており、同一 Docker ネットワーク内のローカルコンテナとのみ通信を行います。
    *   音声認識（Whisper）およびテキスト埋め込み（Embedding）、RAG用のインデックス登録はすべて CPU 推論コンテナ内でローカル完結しています。
*   **外部検索（Brave Search）の検証**:
    *   `.env` 内の `BRAVE_SEARCH_ENABLED` は `false` であり、外部検索APIへの送信コードは明示的に無効化されています。

### 3.2. データベース接続における物理防御の監査
*   **読み取り専用URIの適用**:
    *   `DB_URI` は `sqlite:////data/sqlite/business.db?mode=ro` に設定されており、接続レイヤーで書き込み不可となっています。
    *   万が一LLMが書き込みSQL（例：`UPDATE ...`）を生成し、アプリ層のバリデーションを通過した場合でも、SQLite自体が `attempt to write a readonly database` エラーを発生させてデータを保護します。

### 3.3. SQLバリデーション（ポリシー制御）の監査
`nl2sql_engine.py` 内の `is_safe_query` 関数において、以下の3つの防御壁が設定されています：
1.  **開始文字制限**: クエリの先頭が `SELECT` または `WITH`（共通テーブル式）であることを厳密にチェック。
2.  **禁止コマンドブロック**: 正規表現を用い、以下の操作を大文字小文字を問わず全てブロック：
    *   `insert`, `update`, `delete`, `drop`, `alter`, `create`, `truncate`, `replace`, `grant`, `revoke`, `execute`, `exec`, `commit`, `rollback`
3.  **複数文（セミコロン）実行の遮断**: セミコロン `;` で区切られた複数クエリの検知により、SQLインジェクションによるバックドア作成やデータ消去を防御します。
4.  **リソース保護（自動 LIMIT 追加）**: LIMIT 句が明示されていないクエリには、自動的に `LIMIT 100` が付与され、大量データ引き出しによるメモリ逼迫を防止します。

### 3.4. 監査ログの実装と検証
監査機能は `step2/audit_log/audit_service.py` を用いて、独立したテーブルに永続化されます。
*   **記録されるアクションタイプ**:
    *   `NL2SQL_EXECUTION` (クエリ実行成功)
    *   `NL2SQL_BLOCKED` (セキュリティポリシーによる実行遮断)
    *   `NL2SQL_ERROR` (SQL文法エラーやデータベースエラー)
*   **記録項目**: タイムスタンプ、セッションID、アクション名、ユーザーID（`local_user`）、実行SQL、取得件数、実行所要時間(ms)、エラー詳細。

## 4. 総評
AVITOのセキュリティ設計は、完全閉域での運用要件を十分に満たしており、NL2SQL機能における「物理防御（mode=ro）」と「ポリシー制御（静的チェック・ログ記録）」の二重の防御壁によってデータ破損・漏洩のリスクが最小化されています。本番運用において安全に利用できる状態と判断します。
