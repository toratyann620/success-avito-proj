## ページ 2：エグゼクティブサマリー
### なぜ今、この提案が必要なのか？ そして、どう始めるか！

#### 1. Why なぜ必要か（既存人材の最大化）
- 人材獲得困難な時代に優秀人材は採用しづらく、採用コストも膨大。AI人材はさらに希少。
- 「一人当たりの価値」を「会社の価値」に変えることが、経営の最優先課題。
- 採用・教育に頼り続けるだけでは、組織の競争力強化は間に合わない。今こそ、AIで既存人材の力を最大化する時。

#### 2. What 何が必要か（中位・下位層の底上げ）
- 既存社員の中位6割・下位2割の力をAIで引き出す。（上位2割は自力でAIを活用し成果を出している）
- 組織全体の生産性を決めるのは中位6割・下位2割。この層の能力を引き出す手段は「AI活用」以外にない。
- 今いる人材の底上げが最速の経営改革・組織改革。

#### 3. How どう始めるか（段階的展開）
- AVITOの活用（スタンドアロン型 AI駆動ナレッジ＆データプラットフォーム）。
- 管理部・営業部・技術部・管理職など各社員のPCでナレッジとデータをAI統合活用。
- まずはPoCで効果を実測 → 確認後、段階展開で好循環を生み出す。
- クラウド不使用・完全ローカルで機密情報を守りながら全社展開が可能。
- `PoC → 段階展開 → 全社好循環`

## ページ 8：現場の声が示す課題 ── だからこそ AVITO が必要
### 現場の声（課題）に対し、AVITOが提供する解決策

#### ① スキル格差（使いこなせる人・できない人の差）
- **課題**: チャットベースAIではプロンプトの質で回答精度が変わるため、個人のスキルに依存しやすい。
- **AVITOの解決**: **音声入力対応の業務特化UI** と **承認済み文書参照** で、スキルに関わらず誰でも高品質な資料を生成。属人的依存を排除。

#### ② RAGの精度（データ管理ノウハウが壁になる）
- **課題**: RAG導入を進めても、その土台となるデータ管理に一定のノウハウが必要で精度が上がらない。
- **AVITOの解決**: **承認済み文書のみをインデックス化する自動クローラー** と **業務用語辞書** を標準装備。データ管理ノウハウ不要で高精度RAGが稼働。

#### ③ セキュリティ不安（機密情報の入力範囲が判断困難）
- **課題**: クラウド型チャットAIに機密情報をどこまで入力してよいか判断が難しく、使用が限定的になる。
- **AVITOの解決**: **クラウド不使用の完全ローカル（スタンドアロン）構成** で機密情報を一切外に出さない。最小権限DB接続・監査ログ完備で安全。

## ページ 11：全体アーキテクチャ (Ver0.99)
### クローズドLLM（Open Notebook）をRAG基盤としたアーキテクチャ

1. **要望受付**: 利用者がやりたいことを音声入力（またはキーボード入力）。
2. **PC内部検索**: ローカルPC内の `pptx / docx / xlsx / pdf` を検索。
3. **ファイルサーバ検索**: 社内共有フォルダから関連資料を検索・取り込み（必要があればDB検索）。
4. **WEB検索（指示時）**: 指示があった場合のみ、WEB情報（指定URLも可）から検索・取り込み。
5. **内部資料生成**: 見積書・社内報告書等。内部情報のみを根拠に生成。
6. **外部提供用資料生成**: 提案書等。WEB情報（指定URLも可）も参照。

#### 中核処理基盤
- **Open Notebook（RAGエンジン）**: 文書取り込み、全文検索/ベクトル検索、ノート管理、RAG対話。
- **Ollama（ローカルAIプラットフォーム）**: ローカルLLM実行基盤。
- **Gemma（ローカルLLM）**: 要望解釈 / 要約 / 抽出 / 文書ひな形生成。

#### 運用ルール（出典の分離）
- **内部機密文書モード**: PC内資料 + ファイルサーバ + DB (DB連携は段階的)
- **提案書モード**: PC内資料 + ファイルサーバ + 外部公開情報（WEB検索or指定URL指示時のみ）

## ページ 14：Before / After ── 3階層での効果
### 個人 → 部門 → 会社へ、価値が連鎖する

- **個人 (INDIVIDUAL)**:
  - *Before*: 時間的制限で複数の資料を探しきれず、自分の知っている情報だけで資料を作成している。
  - *After*: 短時間で必要な情報資料をだれでも作成。社内PC・共有サーバ・DBを横断検索。
- **部門 (DEPARTMENT)**:
  - *Before*: 社内PCやサーバに膨大な情報資産が分散し、ファイルを十分活用できていない。
  - *After*: 役職・部門の権限管理に基づき、適切な情報へアクセス可能。情報資産の有効活用。
- **会社 (COMPANY)**:
  - *Before*: 情報の断片化と属人的な知識依存により、組織全体の生産性や意思決定の質がバラバラ。
  - *After*: 情報断片化を解消し、組織全体で均質・高品質な意思決定と資料作成が可能になる。

## ページ 18：スケジュール（案）と役割分担
### AVITO 段階的開発・PoCスケジュール

#### 1. PoC準備フェーズ
- **契約・キックオフ**: 契約書締結・承認、プロジェクト計画書作成。
- **方式設計**: 業務要件・制約の提供、システムアーキテクチャ設計、技術選定。
- **参照資料・データ整備**: 承認済み文書の選定・棚卸、業務定義・業務用語の提供、インデックス要件定義。

#### 2. 開発①フェーズ（AVITO I）
- **コア開発 (DB連携なし)**: ユーザ入力UI開発、ローカルLLM制御部開発、RAGエンジン（クローラー・インデクサ・Web検索）開発、文書生成層・ツール接続開発。
- **動作確認・受入テスト**: テスト環境準備、動作確認。

#### 3. PoC実施フェーズ ① (AVITO I PoC)
- **DB連携なし PoC**: 実ユーザによる試用、効果測定データ収集、フィードバック・改善。

#### 4. 開発②フェーズ（AVITO II）
- **DB連携開発 (会計・販売DB)**: スキーマ/業務定義カタログ構築、NL2SQL変換エンジン開発、SQL制御・実行エンジン開発、監査ログ/実行履歴構築。

#### 5. PoC実施フェーズ ② (AVITO II PoC)
- **DB連携あり PoC**: 実業務データでの試用、SQL精度検証・チューニング、セキュリティ・権限確認。

#### 6. チューニングフェーズ
- **チューニング・改善**: 精度・性能チューニング、ドキュメント・運用保守手順書整備、最終本番展開判断。
