## 1. システムの起動と停止

本システムはすべてのコンポーネントがDockerコンテナ化されています。

### 1.1. システムの起動
プロジェクトのルートディレクトリで以下のコマンドを実行します。
```bash
docker compose up -d
```
*   これにより、`api` (ポート8000), `frontend` (ポート3002), `open-notebook` (ポート5001), `ollama` (ポート11434), `crawler` がバックグラウンドで起動します。
*   初回起動時のみ、`ollama-init` コンテナが走り、モデル（`qwen2.5-coder:1.5b`）の自動ダウンロード（Pull）が行われます。

### 1.2. システムの停止
```bash
docker compose down
```
*   コンテナを終了し、ネットワークを解放します。データベースのデータやLLMモデルデータはボリューム（`sqlite_data`, `ollama_data` 等）に安全に永続化されます。

### 1.3. ログの監視方法
コンテナの動作確認やトラブルシューティング時には、以下のログ確認コマンドを使用します。
```bash
# APIサーバーのログをリアルタイム表示
docker compose logs -f api

# クローラーのファイル検知ログを表示
docker compose logs -f crawler
```

## 2. 初期セットアップとメンテナンス

### 2.1. 模擬ビジネスDBの構築・再作成
テストやデモ用の販売・会計データベース（business.db）を構築、またはリセットしたい場合は以下を実行します。
```bash
docker compose exec api python step2/sample_db_setup.py
```

### 2.2. パフォーマンス最適化（インデックス追加・WALモード適用）
データベースにクエリ最適化インデックスを適用し、WALモード（Write-Ahead Logging）へ切り替える場合は以下を実行します。
```bash
docker compose exec api python step2/add_indexes.py
```

## 3. Ollama モデルのアップグレード・切り替え手順

マシンスペック（GPUの搭載有無、RAM容量）に応じて、利用するLLMを変更することでクエリ生成（NL2SQL）の精度を向上できます。

### 3.1. 推奨モデル基準
*   **メモリ 8GB (CPU推論)**: `qwen2.5-coder:1.5b` (適用中、軽量かつ高速)
*   **メモリ 16GB以上 (CPU/GPU推論)**: `gemma3:4b` または `qwen2.5-coder:7b` (バランス型)
*   **GPU搭載 / メモリ 32GB以上**: `gemma3:12b` もしくは `gemma3:32b` (超高精度・本番推奨)

### 3.2. モデル切り替え方法
1.  **環境変数の書き換え**:
    プロジェクトルートの `.env` ファイル内の `OLLAMA_MODEL` を書き換えます。
    ```ini
    # 例：高精度モデルへ切り替える場合
    OLLAMA_MODEL=gemma3:12b
    ```
2.  **モデルの手動ダウンロード (Pull)**:
    接続先のOllamaコンテナへ新しいモデルをダウンロードするよう指示します。
    ```bash
    docker compose exec ollama ollama pull gemma3:12b
    ```
3.  **コンテナの再起動**:
    変更を適用するため、APIコンテナを再起動します。
    ```bash
    docker compose restart api
    ```

## 4. クローラーの監視設定調整

### 4.1. 監視フォルダの追加・変更
PC内のローカル共有フォルダやファイルサーバーのマウント先を監視対象に加える場合：
1.  `.env` の `WATCH_FOLDERS` を編集します（カンマ区切りで複数指定可能）。
    ```ini
    WATCH_FOLDERS=/watch/pc_local,/watch/file_server
    ```
2.  クローラーを再起動します。
    ```bash
    docker compose restart crawler
    ```

### 4.2. ポーリング間隔（秒）の調整
マウントされたファイルサーバー等へのネットワーク負荷を下げるため、定期的な自動差分スキャンの実行間隔を調整できます。
```ini
# .env ファイル内で秒指定 (例: 600秒 = 10分間隔)
POLL_INTERVAL_SECONDS=600
```

## 5. トラブルシューティング

### 5.1. ポートの競合エラーで起動しない
*   **原因**: macOSのAirPlayサービスがポート `5000` を占有している、または他のNode.jsプロセスが `3000` を使用している。
*   **対策**: 本システムでは、Open Notebookの公開ポートを `5001` に、フロントエンドNext.jsの公開ポートを `3002` に設定してポート競合を回避しています。競合が再発した場合は、`docker-compose.yml` 内の `ports` 定義（左側のホストポート番号）を別の値に変更してください。

### 5.2. LLMの応答がタイムアウトする（または返ってこない）
*   **原因**: ローカルCPUでの推論負荷が一時的に高くなり、処理がタイムアウト値を超えた。
*   **対策**:
    *   `.env` 内の `OLLAMA_TIMEOUT` を大きめの値（例: `300` 秒）に引き上げます。
    *   より軽量なモデル（`qwen2.5-coder:1.5b`）に変更されていることを確認します。
    *   Docker Desktopへのメモリ割り当てを、環境設定から可能な限り拡張（例: 4GB以上）してください。
