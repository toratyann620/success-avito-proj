---

## 1. 開発方針・原則

### 基本方針
| 原則 | 内容 |
|-----|------|
| **安全第一** | 社内情報のクラウド送信ゼロ。すべての処理はローカルPC内で完結させる |
| **段階的デリバリー** | Step1（ローカル完結）を先に完成させ、Step2（DB連携）へ段階的に拡張 |
| **Docker必須** | 全コンポーネントをDockerコンテナで動作させ、環境依存を排除 |
| **再現性** | どの環境でも `docker compose up` 一発で起動できる構成とする |
| **品質** | 各フェーズ完了時にE2Eテストを実施し、次フェーズへ進む |

## 5. リスク管理

| リスク | 影響度 | 対策 |
|--------|--------|------|
| Gemma3:12bの日本語精度が不足 | 高 | Qwen2.5:14b、llama3.1:8bをバックアップとして検証 |
| Docker Desktopのスペック不足（VRAM） | 高 | GPU対応確認、CPU推論モードへの切り替え手順を整備 |
| Open Notebookの日本語対応の限界 | 中 | カスタムRAGパイプラインに切り替える判断基準を事前定義 |
| NL2SQLの精度（Step2） | 高 | 業務定義カタログの品質が鍵。お客様と事前の業務用語ヒアリングを必須とする |
| ファイルサーバへのアクセス権限 | 中 | 読み取り専用アクセス権の設定をお客様IT部門と事前調整 |

## 7. 定義：完了基準（Definition of Done）

各フェーズが「完了」となる基準：

| フェーズ | 完了基準 |
|---------|---------|
| Phase 0 | `docker compose up` で全コンテナが起動し、Gemma3:12bへの日本語問い合わせが成功すること |
| Phase 1 | サンプルドキュメント（Word/Excel/PDF各1件）を取り込み、日本語で質問して出典付き回答が返ること |
| Phase 2 | 音声入力 → RAG検索 → Excel/Word/PPT生成の全フローが1回のオペレーションで完了すること |
| Phase 3 | 自然言語入力からSQLが生成・実行され、結果が正しく返ること（テストクエリ50問中40問以上） |
| Phase 4 | Step1 + Step2 統合環境が安定稼働し、全ドキュメントが納品できること |
