## 1. 開発方針・原則

### 基本方針
| 原則 | 内容 |
|:---|:---|
| **安全第一** | 機密情報・社内情報のクラウド送信ゼロ。すべての処理は完全ローカル環境で完結させる（閉域網運用）。 |
| **段階的デリバリー** | DB連携なしの「AVITO I」でPoCを実施し、その後DB連携ありの「AVITO II」へ段階的に拡張。 |
| **Docker必須** | 全コンポーネントを Docker Desktop 上のコンテナ環境で動作させ、実効環境を分離・ポータブルにする。 |
| **再現性** | どの環境でも `docker compose up` 一発で起動・稼働できるシンプルなコンテナ構成とする。 |
| **品質** | 各フェーズでの品質検証ロジックを組み込み、ハルシネーション抑制と出力の安定性を保証する。 |

## 3. フェーズ詳細

### Phase 1：PoC準備 & 資料整備
**目標**: 方式設計を確定し、RAG対象の「承認済み文書・完成版文書」の棚卸と、業務用語・KPIの定義を整備する。

*   **タスク**:
    *   既存システム構成情報の共有、アーキテクチャレビュー。
    *   インデックス対象の要件定義、フォルダ構成ルールの整備（完成版のみをインデックス化する仕組みの設計）。
    *   業務用語・KPI定義の提供、業務用語辞書の基本設計。

---

### Phase 2：開発① ── AVITO I コア開発 (DB連携なし)
**目標**: 本プロジェクトのフロントエンドUIをデモ（050）と完全に統合し、ローカルPC内のファイルを高精度に検索・RAG回答できる基盤を構築する。

*   **タスク**:
    *   **フロントエンド完全移植**: Next.js + framer-motion + lucide-react の完全移植（完了）。
    *   **PC内部クローラー実装**: watchdog + SQLite FTS5 によるローカルファイル自動インデックス監視モジュールの実装・検証（完了）。
    *   **RAGエンジンの構築**: Open Notebook（LlamaIndex等）を利用した全文検索・ベクトル検索の統合。
    *   **文書生成モジュールの統合**: Word (`python-docx`), Excel (`openpyxl`), PowerPoint (`python-pptx`) による文書自動生成。
    *   **外部Web検索の統合**: Brave Search API を用い、指示時のみ外部情報を取得し「出典を分離して表示」する機能の実装。

---

### Phase 3：PoC実施① ── AVITO I による検証
**目標**: DB連携なしの状態で実ユーザに試用してもらい、UI/UXおよびRAGの検索・生成精度について効果を測定・改善する。

*   **タスク**:
    *   PoC環境の構築・運用支援。
    *   ユーザフィードバックの収集・優先度判定・改善対応。
    *   「見積書作成」「報告書比較」など主要ユースケースの稼働検証。

---

### Phase 4：開発② ── AVITO II DB連携開発
**目標**: 自然言語入力から安全にSQLを生成・実行（NL2SQL）し、会計DB・販売DBのデータを分析・帳票化するモジュールを開発する。

*   **タスク**:
    *   **業務定義カタログ構築**: テーブル定義、項目定義、業務用語マッピング。
    *   **NL2SQL変換エンジン**: Gemmaによる安全なSQL自動生成、禁止SQL（DROP/UPDATE等）チェック機能の実装。
    *   **DB接続ブリッジ**: SQLAlchemyを使用した「読み取り専用接続（書き込み権限なし）」の構築。
    *   **監査ログ機能**: 実行SQL・実行者・時刻の記録。

---

### Phase 5：PoC実施② ── AVITO II による検証
**目標**: 会計・販売DBの実際のデータを用いて検証し、SQLの生成精度およびセキュリティに問題がないか確認する。

*   **タスク**:
    *   実業務データを用いた試用とSQL精度検証・チューニング。
    *   アクセス権・権限設定の監査ログ確認。

---

### Phase 6：チューニング & 最終改善
**目標**: 精度・パフォーマンスの最大化、運用保守手順書の整備、本番展開に向けた最終確認を行う。

## 5. 想定リスクと対策

最新提案書 (Ver0.99) に基づき、以下の5つの区分でリスク管理を徹底する。

### ① 回答・生成品質リスク
*   **ハルシネーション**: 根拠のない回答の出力。
    *   *対策*: **承認済み文書のみをRAGの参照対象** とし、プロンプトを固定化。出力検証ロジックを実装。
*   **テンプレート崩れ**: 自動生成されたExcelやWordのフォーマット破損。
    *   *対策*: 事前に固定されたテンプレートファイルをベースとし、データの差し込み・グラフの追加のみをライブラリ経由で行うことでフォーマットを保護。

### ② ナレッジ・データ品質リスク
*   **未承認文書の混入**: ドラフトや古い資料の参照による誤回答。
    *   *対策*: クローラーのインデックス対象を「完成版・提出済み」の特定フォルダのみに制限（**完成版のみインデックス化**）。
*   **用語定義の不一致**: 業務用語とDB項目の乖離による誤SQL生成。
    *   *対策*: 業務用語辞書の整備と、テーブル・項目をマッピングする「業務定義カタログ」を事前に構築・更新。

### ③ セキュリティ・権限リスク
*   **権限外文書の参照 / 情報漏洩**:
    *   *対策*: アクセス権の継承管理、および外部Web検索はデフォルトでOFFとし指示時のみ明示的に使用。
*   **過剰権限DB接続**: NL2SQLによる不正操作。
    *   *対策*: DB接続は **読み取り専用（書き込み権限なし）** に制限。高負荷クエリや禁止クエリ（DELETE/DROPなど）の事前検知ロジックを搭載。監査ログを完備。

### ④ システム運用・性能リスク
*   **端末高負荷 / レスポンス遅延**: ローカルPC動作での高負荷。
    *   *対策*: クローラーは watchdog を用いた差分更新＋ハッシュ管理とし、常時スキャンを回避。DB連携においては夜間バッチ等で最適化されたデータマートへの転送を推奨。

### ⑤ 業務運用・統制リスク
*   **保管ルール未整備 / 承認区分の混在**:
    *   *対策*: ファイルサーバ・PC内のファイル命名規則の統一、フォルダ管理による承認区分の明示管理、部門横断レビューによる用語差異の解消。

## 6. 定義：完了基準（Definition of Done）

*   **PoC準備フェーズ**: 方式設計書と参照資料の棚卸・整備ガイドラインが完成し、合意されていること。
*   **開発① (AVITO I) フェーズ**: ローカルPC内の対象フォルダ監視・FTS5登録が行われ、Next.js UIから音声/キーボードで入力した質問に対して、PC内資料に基づく「出典付き回答」および「Excel/Word/PPTドラフト」が正常に生成されること。
*   **開発② (AVITO II) フェーズ**: 読み取り専用DBブリッジ経由でSQLが生成・実行され、取得データに基づき数値の差異検証が自動で実行され、正しい帳票が生成されること。
